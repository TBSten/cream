# Managing UI state with sealed classes and cream.kt

In declarative UI / UDF (Unidirectional Data Flow) architectures, it is idiomatic to model a screen's state as a `sealed interface` (or `sealed class`) UiState — `Loading` / `Success` / `Error` become distinct types, `when` branches are exhaustively checked by the compiler, and impossible state combinations are ruled out at compile time.

The trouble starts with the **shared properties** these states carry. Values like "which item is being displayed" must survive every state transition, so every transition ends up hand-copying them from the previous state. cream.kt generates exactly this carry-over code for you, so each transition only has to state *what actually changed*.

This page condenses the key ideas of the five-part series into one self-contained overview. The full articles are linked at the bottom.

## The problem: hand-carrying shared properties

Suppose an item-detail screen models its state like this, with `itemId`, `isBookmarked`, and `snackbarMessage` declared on the sealed parent and overridden by every subtype:

```kt
sealed interface ItemDetailUiState {
    val itemId: String
    val isBookmarked: Boolean
    val snackbarMessage: String?

    // each subtype overrides all three shared properties (abbreviated as /* ... */)
    data class Loading(override val itemId: String, /* ... */) : ItemDetailUiState
    data class Success(override val itemId: String, val item: Item, /* ... */) : ItemDetailUiState
    data class Error(override val itemId: String, val message: String, /* ... */) : ItemDetailUiState
}
```

A `Loading → Success` transition written by hand has to carry every shared property across:

```kt
fun onItemLoaded(prev: ItemDetailUiState, item: Item): ItemDetailUiState =
    ItemDetailUiState.Success(
        itemId = prev.itemId,
        item = item,
        isBookmarked = prev.isBookmarked,       // this line multiplies with every transition
        snackbarMessage = prev.snackbarMessage, // this line multiplies with every transition
    )
```

The only meaningful change is `item = item`; everything else is carry-over noise. Add one more shared property and **every transition call site** in the codebase needs another line. Worse, forgetting `prev.isBookmarked` and writing a fixed value like `isBookmarked = false` still compiles, so the bug slips through review.

## `@CopyToChildren`: transitions between states, minus the boilerplate

Annotate the sealed parent once:

```kt
@CopyToChildren
sealed interface ItemDetailUiState { /* ... */ }
```

cream.kt generates a copy extension function from the parent type to **every transitive concrete leaf**. Shared properties whose names match the parent's get `= this.xxx` defaults:

```kt
fun ItemDetailUiState.copyToItemDetailUiStateSuccess(
    itemId: String = this.itemId,
    isBookmarked: Boolean = this.isBookmarked,
    snackbarMessage: String? = this.snackbarMessage,
    item: Item,
): ItemDetailUiState.Success
```

The transition collapses to its essence:

```kt
fun onItemLoaded(prev: ItemDetailUiState, item: Item): ItemDetailUiState =
    prev.copyToItemDetailUiStateSuccess(item = item)
```

Adding a new shared property later changes the sealed declaration only — **no transition call site is touched**, because the generated defaults pick it up automatically.

This also scales to nested state machines. Because `@CopyToChildren` recurses through intermediate sealed types, a multi-phase checkout flow (`CheckoutUiState.Cart.Editing`, `CheckoutUiState.Payment.Processing`, …) is fully covered by a single annotation on the root — no per-leaf `@CopyTo`, and new leaves are picked up automatically (see [Part 3](./03.md)).

## `@SealedCopy`: updating shared properties while keeping the subtype

Pull-to-refresh and optimistic updates need the opposite move: keep the current subtype and its data, and change only a shared property. Since `data class` `copy` is only available once the subtype is known, doing this on a parent-typed `state: FeedUiState` normally forces a `when` over all subtypes:

```kt
fun FeedUiState.startRefreshing(): FeedUiState = when (this) {
    is FeedUiState.Loading -> this // object: nothing to update
    is FeedUiState.Content -> this.copy(isRefreshing = true) // posts kept as-is
    is FeedUiState.Error -> this.copy(isRefreshing = true)   // throwable kept as-is
}
```

This same-shaped `when` multiplies for every shared-property tweak (finish refreshing, show banner, dismiss banner, …), and every new subtype must be added to all of them. `@SealedCopy` on the sealed parent generates that `when` for you, exposed as a parent-typed `copy()`:

```kt
// generated by @SealedCopy(nonCopyableStrategy = NonCopyableStrategy.RETURN_AS_IS)
fun FeedUiState.copy(
    isRefreshing: Boolean = this.isRefreshing,
    bannerMessage: String? = this.bannerMessage,
): FeedUiState = when (this) {
    is FeedUiState.Loading -> this // non-copyable object: returned as-is (RETURN_AS_IS)
    is FeedUiState.Content -> this.copy(isRefreshing = isRefreshing, bannerMessage = bannerMessage)
    is FeedUiState.Error -> this.copy(isRefreshing = isRefreshing, bannerMessage = bannerMessage)
}
```

Callers update shared properties in one line, without knowing or caring which subtype is current — `Content` keeps its `posts`, `Error` keeps its `throwable`:

```kt
val startRefreshing = state.copy(isRefreshing = true)
val finishRefreshing = state.copy(isRefreshing = false)
```

Non-copyable subtypes such as `data object Loading` are handled by `nonCopyableStrategy` (`ERROR` by default, or `RETURN_AS_IS` / `RETURN_NULL`), so nothing falls back silently.

## Putting it together: a declarative MVI reducer

The two sealed-copy annotations compose. With `@CopyToChildren` handling transitions *between* states and `@SealedCopy` handling shared-property updates *within* a state, an MVI reducer reads as a declaration of per-event diffs:

```kt
fun CounterUiState.reduce(event: CounterEvent): CounterUiState = when (event) {
    CounterEvent.Start        -> copyToCounterUiStateCounting(count = 0)
    is CounterEvent.Increment -> if (this is CounterUiState.Counting) copy(count = count + event.by) else this
    CounterEvent.Finish       -> if (this is CounterUiState.Counting) copyToCounterUiStateFinished(total = count) else this
}
```

The shared context (`userId`, `sessionStartedAt`, …) never appears: each branch states only `count = 0`, `count + event.by`, or `total = count` — the actual specification of the state transition. However many shared properties are added later, this reducer does not change by a single line.
